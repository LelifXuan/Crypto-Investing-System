# 来源选择说明

## 1. 总体原则

这批来源分四层：

- **第 1 层：官方宏观与监管源**
  - 国家统计局
  - 中国人民银行
  - 国家外汇管理局
  - 中国政府网
  - 中国证监会
  - 上交所

- **第 2 层：国内财经日历聚合**
  - 金十财经日历
  - 东方财富财经日历

- **第 3 层：交易所与项目事件**
  - Gate Announcements
  - Gate New Listings
  - Gate Crypto Calendar

- **第 4 层：Web3 媒体与链上供应商**
  - PANews RSS
  - Odaily 快讯
  - Tokenview API

## 2. 为什么这样分层

### 官方层
适合：
- 宏观数据发布时间表
- 政策发布
- 监管口径
- 中国本土市场制度变化

特点：
- 可信度高
- 频率低
- 结构未必规范
- 更适合做“高优先级事件源”

### 国内聚合层
适合：
- 全球宏观事件日历
- 休市安排
- 会议日历
- 重要数据发布提醒

特点：
- 可用性强
- 结构对交易更友好
- 适合作为官方源的日历补充和 fallback

### 交易所/项目层
适合：
- 上线/下线
- 维护
- 代币解锁
- 项目升级
- 上币和合约开通

特点：
- 与你的 Gate 行情链路天然匹配
- 对加密交易实用价值最高
- 应该作为高频轮询任务

### Web3 媒体/链上层
适合：
- 快讯流
- 热点事件补充
- 链上异动监控
- 巨鲸/交易所流向/高额转账

特点：
- 速度快
- 噪音相对更高
- 必须做去重、可信度分层和告警降噪

## 3. 不建议首批接入的来源

以下来源不建议放在“国内不翻墙优先”的首批包里：
- 依赖海外 API Key 且经常限流/付费的国外日历服务
- 需要稳定国际网络的媒体 RSS
- 大量社交媒体源（X/Telegram）直抓
- Glassnode / Santiment / Nansen 这类更适合作为指标源而不是事件主源的付费服务

## 4. 事件模型建议

不要把所有外部消息都视为同一种 event。建议至少拆成：

- `calendar_event`
- `release_event`
- `policy_event`
- `exchange_event`
- `project_event`
- `newsflash_event`
- `onchain_alert_event`

然后再统一投影到 `market_events` 表。

## 5. 告警建议

### 必须实时告警
- Gate 下架/维护/暂停提现
- 央行公开市场业务规则调整
- LPR 发布
- PMI / PPI / GDP / 国民经济运行情况
- Token 解锁（高市值或你持仓标的）
- 链上大额转账 / 交易所净流入暴增

### 应该低频聚合后告警
- 媒体快讯
- 品牌活动
- 一般会议/论坛
- 一般政策解读

## 6. 实现建议

- 用统一的 `SourceAdapter` 抽象
- 对 HTML 源提供：
  - requests 直抓
  - Playwright fallback
- 对 API 源提供：
  - key 管理
  - 限流器
  - 分页器
- 对 RSS 源提供：
  - feedparser 适配层
- 所有 source 输出统一的 `RawEventCandidate`
